# Lark TODO Bot 設計文件

> 日期：2026-03-16
> 狀態：設計階段

## 1. 概述

一個運行在 Lark 上的個人 TODO 管理 Bot，核心功能：

1. **TODO 管理** — 透過 Bot 對話新增、查看、完成、取消工作項目
2. **自動請假** — 偵測到「請假」項目時，自動詢問是否建立 Lark 審批單
3. **請假群組通知** — 假期前兩天自動在指定群組通知全體成員
4. **每日工作提醒** — 每天早上 Bot 私訊當天的 TODO 清單

**時區：** 所有時間皆以 Asia/Taipei（UTC+8）為準。

### 核心元件

| 元件 | 職責 |
|------|------|
| **Lark Bot** | 接收使用者指令（新增/查看/刪除 TODO），回覆互動 |
| **Bitable** | 儲存 TODO 資料（日期、內容、是否為請假、事由等） |
| **排程器** | 每天早上發 TODO 提醒、每天檢查後天是否有請假並通知群組 |
| **Approval 模組** | 偵測到「請假」TODO 時，詢問使用者是否建立審批單 |
| **IM 通知模組** | 發送私訊提醒、群組通知 |

### 資料流

1. **新增 TODO** → 使用者跟 Bot 說 → Bot 寫入 Bitable → 若內容含「請假」→ Bot 回問「是否建立假單？」
2. **每日提醒** → 排程器每天早上讀取 Bitable 當天 TODO → Bot 私訊使用者
3. **請假通知** → 排程器每天檢查後天是否有請假 TODO → 有的話 Bot 發訊息到指定群組

## 3. Bitable 資料結構

### TODO 主表

| 欄位名稱 | 類型 | 說明 |
|----------|------|------|
| `id` | 自動編號 | 唯一識別碼，用於指令中引用（如 `完成 3`） |
| `date` | 日期 | TODO 對應的日期 |
| `content` | 文字 | 工作項目內容 |
| `is_leave` | 布林 | 是否為請假項目 |
| `leave_reason` | 文字 | 請假事由（僅 is_leave=true 時有值） |
| `leave_type` | 單選 | 請假類別（事假/病假/特休等） |
| `leave_group_id` | 文字 | 多天請假的關聯 ID（同一次請假共用同一個 UUID） |
| `approval_status` | 單選 | 審批狀態：未建立 / 已送出 / 已通過 / 已拒絕 |
| `approval_id` | 文字 | Lark 審批單 ID（建立後回填） |
| `notified` | 布林 | 是否已發送群組通知（避免重複通知） |
| `status` | 單選 | TODO 狀態：待辦 / 完成 / 取消 |
| `created_at` | 日期時間 | 建立時間 |

**`id` 欄位說明：** Bitable 自動編號為連續遞增整數。Bot 在回覆清單時會顯示此 `id`，使用者使用 `完成 <id>` / `取消 <id>` 時即引用此值。Bot 內部以 Bitable record_id 做實際操作，`id` 僅作為使用者面向的顯示用途。

**`leave_group_id` 欄位說明：** 當請假跨多天時（如 3/21~3/23），拆成多筆 TODO 記錄，但共用同一個 `leave_group_id`。群組通知時依此欄位合併為一則通知。審批單也只建立一筆，所有相關記錄共用同一個 `approval_id`。

### 設定表（Config）

存放個人設定，僅一筆資料。若資料遺失，系統啟動時自動以環境變數預設值重建。

| 欄位名稱 | 類型 | 說明 |
|----------|------|------|
| `user_id` | 文字 | Lark User ID |
| `notify_group_id` | 文字 | 請假通知的目標群組 ID |
| `morning_remind_time` | 文字 | 每日提醒時間（預設 09:00） |
| `notify_days_before` | 數字 | 提前幾天通知（預設 2） |
| `approval_code` | 文字 | 公司請假審批表的 approval_code |

## 4. Bot 指令與互動流程

### 日期格式

支援以下格式，年份省略時預設為當年（若日期已過則自動推到明年）：

| 格式 | 範例 |
|------|------|
| `M/D` | `3/20` |
| `M月D日` | `3月20日` |
| `YYYY/M/D` | `2026/3/20` |
| `今天` / `明天` / `後天` | |
| `下週一` ~ `下週日` | |
| `M/D~M/D`（區間） | `3/21~3/23`（用於多天請假） |

### 支援的指令

| 指令 | 說明 | 範例 |
|------|------|------|
| `新增 <日期> <內容>` | 新增一筆 TODO | `新增 3/20 準備週會簡報` |
| `請假 <日期> <事由>` | 新增請假 TODO（自動標記 is_leave） | `請假 3/21 家中有事` |
| `今天` | 查看今天的 TODO 清單 | `今天` |
| `本週` | 查看本週的 TODO 清單 | `本週` |
| `完成 <id>` | 將某筆 TODO 標記為完成 | `完成 3` |
| `取消 <id>` | 取消某筆 TODO | `取消 5` |
| `建立假單 <id>` | 對未建立假單的請假項目補建審批單 | `建立假單 7` |
| `設定群組` | 設定請假通知的目標群組 | （Bot 引導操作） |
| `設定提醒 <時間>` | 設定每日提醒時間 | `設定提醒 08:30` |
| `help` / `?` / `幫助` | 顯示所有可用指令說明 | `help` |

### 錯誤處理

Bot 指令執行失敗時（如 Bitable 寫入失敗、API 逾時），Bot 回覆簡短錯誤訊息：

```
Bot: ❌ 操作失敗，請稍後再試。若持續發生請檢查 Bot 服務狀態。
```

不做自動重試，由使用者決定是否重新執行指令。

## 5. 排程與通知邏輯

### 排程任務

所有排程時間以 Asia/Taipei（UTC+8）為準。Cron 表達式需換算為 UTC。

| 排程 | 預設時間 | 邏輯 |
|------|---------|------|
| **每日 TODO 提醒** | 每天 09:00 | 查詢 Bitable 當天的 TODO → Bot 私訊使用者 |
| **請假群組通知** | 每天 10:00 | 查詢 Bitable 後天是否有 `is_leave=true` 且 `notified=false` 的項目 → 發送群組通知 → 標記 `notified=true` |

兩個排程分開時間點：09:00 先收到自己的 TODO 提醒，10:00 才發群組通知，避免訊息混在一起。

### Render 免費方案排程可靠性

Render 免費方案閒置 15 分鐘後休眠，cron 排程將無法觸發。**必須搭配外部保活服務：**

- 使用 [UptimeRobot](https://uptimerobot.com/)（免費）每 5 分鐘 ping 服務的 `/health` endpoint
- 服務提供 `GET /health` 回傳 200，確保不被休眠

這是必要設定，非可選項。

### 每日 TODO 提醒流程

```
[09:00 排程觸發]
    │
    ▼
查詢 Bitable: date = 今天 AND status = 待辦
    │
    ├── 有資料 → 組裝 Message Card 私訊使用者
    │            （包含所有待辦項目，請假項目特別標注）
    │
    └── 無資料 → 不發送（避免打擾）
```

### 請假群組通知流程

```
[10:00 排程觸發]
    │
    ▼
查詢 Bitable: date = 後天 AND is_leave = true AND notified = false
    │
    ├── 有資料 → 依 leave_group_id 合併同一次請假：
    │            1. 發送群組通知 Message Card（多天請假合併為一則）
    │            2. 更新所有相關記錄 notified = true
    │
    └── 無資料 → 不執行
```

### 請假審批單建立流程

```
[使用者輸入「請假」指令 或 新增 TODO 偵測到「請假」關鍵字]
    │
    ▼
寫入 Bitable（is_leave=true）
    │
    ▼
Bot 詢問假別 → 使用者點選
    │
    ▼
Bot 詢問「是否建立假單？」
    │
    ├── [確認送出]
    │     │
    │     ▼
    │   呼叫 Approval API 建立審批實例
    │     │
    │     ├── 成功 → 回填 approval_id，更新 approval_status = 已送出
    │     │
    │     └── 失敗 → 通知使用者錯誤原因，approval_status 維持未建立
    │
    └── [稍後再說]
          │
          ▼
        approval_status 維持「未建立」
        （使用者可用 `建立假單 <id>` 指令補建）
```

### 審批狀態回調

訂閱 Lark 事件 `approval.instance.status_changed`，當審批單狀態變更時：

1. 收到回調事件
2. 依 `approval_id` 找到對應的 Bitable 記錄
3. 更新 `approval_status` 為「已通過」或「已拒絕」
4. Bot 私訊通知使用者審批結果

### 邊界處理

| 情境 | 處理方式 |
|------|---------|
| 請假跨多天（如 3/21~3/23） | 拆成多筆 TODO（共用 `leave_group_id`），建立一筆審批單，群組通知合併為一則 |
| 使用者取消請假 TODO | 若假單已送出，提示使用者需到 Lark 審批中手動撤回 |
| 排程執行時 API 失敗 | 不更新 `notified`，下次排程會重試 |
| 假期就是明天或今天才新增 | 立即發送群組通知，不等排程 |

## 6. 技術棧

| 層面 | 選擇 | 原因 |
|------|------|------|
| **語言** | TypeScript | 型別安全、Lark SDK 支援良好 |
| **執行環境** | Node.js | 輕量、非同步處理適合 Bot 場景 |

## 8. 安全性

### Webhook 驗證

所有來自 Lark 的 Webhook 請求必須驗證真實性：

- 使用 Lark App 的 **Verification Token** 驗證事件回調
- 使用 **Encrypt Key** 解密加密的事件內容（若啟用）
- 拒絕所有無法通過驗證的請求

Lark 官方 SDK `@larksuiteoapi/node-sdk` 內建了驗證機制，正確初始化即可。

## 9. 部署

### 目標平台：Render（免費方案）

- **服務類型：** Web Service
- **公網 URL：** Render 自動提供，用於 Lark Bot Webhook 回調
- **保活：** 必須設定 UptimeRobot 每 5 分鐘 ping `/health` endpoint

### 環境變數

| 變數名 | 說明 |
|--------|------|
| `LARK_APP_ID` | Lark App ID |
| `LARK_APP_SECRET` | Lark App Secret |
| `LARK_VERIFICATION_TOKEN` | Webhook 驗證 Token |
| `LARK_ENCRYPT_KEY` | 事件加密金鑰（選填） |
| `BITABLE_APP_TOKEN` | Bitable 多維表格的 App Token |
| `BITABLE_TODO_TABLE_ID` | TODO 主表的 Table ID |
| `BITABLE_CONFIG_TABLE_ID` | Config 設定表的 Table ID |
| `DEFAULT_USER_ID` | 使用者的 Lark User ID |
| `DEFAULT_NOTIFY_GROUP_ID` | 預設通知群組 ID |
| `DEFAULT_APPROVAL_CODE` | 請假審批表代碼 |
| `TZ` | 時區，設為 `Asia/Taipei` |

### Lark 後台設定

| 項目 | 說明 |
|------|------|
| 建立 Lark App | 取得 App ID + App Secret |
| 啟用 Bot 功能 | 讓 App 可作為 Bot 接收訊息 |
| 設定事件訂閱 | 訂閱 `im.message.receive_v1`（接收訊息）及 `approval.instance.status_changed`（審批狀態變更） |
| 申請權限 | `bitable:record:read`、`bitable:record:write`、`im:message:send`、`approval:instance:create`、`approval:instance:read`、`contact:user.id:readonly` |
| 取得 approval_code | 從審批管理找到請假表單的代碼 |
